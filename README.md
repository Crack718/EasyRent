# EasyRent

EasyRent — это веб-приложение для аренды жилья на `Vanilla JS + Firebase`.
Проект показывает полный цикл работы сервиса аренды: каталог, карточка объявления, бронирование, личный кабинет, админ-панель, real-time обновления и ролевой доступ.

## Зачем нужен этот сайт

EasyRent нужен как готовая база для:

- MVP сервиса аренды (краткосрочной или долгосрочной).
- Демо/учебного проекта по Firebase (Firestore, Auth, Rules, индексы, realtime).
- Быстрого старта собственного продукта с уже реализованными ключевыми сценариями.

## Как работает сайт

### Пользовательский поток

1. Пользователь открывает каталог на главной странице.
2. Ищет жильё по тексту, фильтрует по параметрам, сортирует и листает страницы кнопкой «Показать ещё».
3. Переходит в карточку объявления, смотрит фото, отзывы, доступность и похожие варианты.
4. Оставляет сообщение владельцу, добавляет в избранное, создаёт бронирование.
5. В личном кабинете видит профиль, избранное, историю, отзывы и сообщения.
6. В разделе бронирований управляет активными действиями и историей.

### Админский поток

1. Администратор заходит в `admin.html` (доступ только при роли `admin`).
2. Управляет объявлениями и пользователями (CRUD).
3. Модерирует сообщения/отзывы и меняет статусы бронирований.
4. Видит базовую статистику по проекту.

## Реализованные страницы

- `index.html` — каталог, поиск, фильтры, сортировка, пагинация, real-time.
- `listing.html` — детали объявления, галерея, отзывы, сообщения, форма бронирования, похожие объявления.
- `profile.html` — профиль, избранное, история бронирований, мои отзывы, мои сообщения.
- `cart.html` — активные и завершённые бронирования, действия (изменить/подтвердить/отменить/завершить/удалить).
- `bookings.html` — редирект на `cart.html`.
- `admin.html` — админ-панель с управлением сущностями и модерацией.

## Основные функции

- Аутентификация: регистрация, вход, выход, восстановление пароля.
- Роли: `user` и `admin`.
- Real-time через `onSnapshot` (каталог, объявления, бронирования, сообщения, отзывы).
- Пагинация и догрузка (`limit`, `startAfter`, кнопка «Показать ещё»).
- Поиск и фильтрация по параметрам жилья.
- Сортировка по дате, цене и рейтингу.
- Избранное пользователя.
- Бронирования с проверкой пересечений дат.
- Денормализация карточек в `listing_cards` для быстрых запросов.
- Светлая/тёмная тема + анимации интерфейса.

## Технологии

- `HTML5`, `CSS3`, `Vanilla JavaScript (ES6 modules)`.
- `Firebase Authentication`.
- `Cloud Firestore`.
- `Firebase Security Rules`.
- `Firebase Indexes`.
- Дополнительно: `Cloud Functions` (HTTP endpoint `catalogQuery`, опционально).

## Firestore: модель данных

Основные коллекции:

- `users`: профиль пользователя (`name`, `email`, `phone`, `role`, `createdAt`).
- `listings`: полные объявления.
  - Подколлекция `images`: изображения объявления.
  - Подколлекция `reviews`: отзывы.
- `listing_cards`: облегчённые карточки для каталога (денормализация из `listings`).
- `amenities`, `regions`, `districts`, `property_types`: справочники фильтров.
- `favorites`: связь пользователь ↔ объявление.
- `messages`: сообщения между пользователем и владельцем.
- `bookings`: все бронирования.
- `availability`: активные занятые слоты по объявлениям.
- `active_actions`: активные действия пользователя.
- `action_history`: завершённые/отменённые действия.

## Безопасность (Rules)

`firestore.rules` реализует:

- Ограничение доступа по ролям (`user/admin`).
- Доступ к админским операциям только для `admin`.
- Пользователь может редактировать только свой профиль и свои данные.
- Ограничения на изменение сообщений и бронирований (только допустимые поля).
- Публичное чтение каталога, защищённая запись.

## Индексы Firestore

В проекте уже есть `firestore.indexes.json` для:

- Каталога (статус + сортировка + фильтры).
- Поиска по `searchTokens`.
- Сообщений.
- Бронирований.
- `listing_cards`.

Если видите ошибку `The query requires an index`, нужно заново задеплоить индексы.

## Локальный запуск

### 1) Установка зависимостей

```powershell
npm install
```

### 2) Запуск как статического сайта

Подойдёт любой локальный сервер (например Live Server, `http-server`, `serve`).

Пример:

```powershell
npx http-server . -p 5173
```

Откройте:

- [http://127.0.0.1:5173/index.html](http://127.0.0.1:5173/index.html)

## Firebase: подготовка проекта

1. Создайте проект в Firebase Console (или используйте существующий).
2. Включите `Authentication` (Email/Password).
3. Создайте базу `Firestore`.
4. Проверьте `window.EASYRENT_CONFIG` в HTML-страницах (ключи проекта).
5. Задеплойте правила и индексы:

```powershell
firebase deploy --only firestore --project easyrent-369d9
```

или отдельно:

```powershell
firebase deploy --only firestore:rules --project easyrent-369d9
firebase deploy --only firestore:indexes --project easyrent-369d9
```

## Демо-данные

В проекте есть утилиты для наполнения/очистки данных:

- `seedTestData()` в клиентском коде (через админ-панель).
- `scripts/reset-demo.js` — пересоздаёт демо-набор.
- `scripts/sync-listing-cards.js` — синхронизирует `listing_cards` с `listings`.

Запуск утилит через сервисный аккаунт:

```powershell
$env:GOOGLE_APPLICATION_CREDENTIALS="C:\path\to\service-account.json"
node scripts/reset-demo.js
node scripts/sync-listing-cards.js
```

## Проверка работоспособности

Автопроверки UI/консоли:

```powershell
node scripts/playwright-check.js
node scripts/index-ui-check.js
node scripts/loadmore-check.js
```

Скрипты проверяют:

- загрузку страниц;
- базовые элементы UI;
- ошибки в консоли/сети;
- корректность кнопки «Показать ещё».

## Разработка и архитектурные решения

- Проект разделён на уровни:
  - `js/pages/*` — логика конкретных страниц.
  - `js/services/*` — работа с Firestore.
  - `js/core/*` — инициализация Firebase/Auth.
  - `js/ui/*` — рендер HTML.
  - `js/utils/*` — общие хелперы.
- Каталог использует `listing_cards`, чтобы снизить стоимость и сложность запросов.
- При ошибках индекса в каталоге есть fallback на `listings` (временная мера, пока индекс строится).
- Бронирования зеркалятся в `active_actions`, `action_history`, `availability` для удобных выборок.

## Что важно для production

- Убрать публичные конфиги из HTML и хранить их через безопасную сборку/переменные окружения.
- Подключить CI/CD и линтинг.
- Вынести чувствительные операции в Cloud Functions.
- Добавить полноценные e2e/интеграционные тесты и мониторинг ошибок.
